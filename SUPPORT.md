# Support

Use the right channel so maintainers can respond without losing important
context.

## Questions and usage help

For setup questions, API usage, provider configuration, or integration advice,
open a GitHub Discussion when Discussions are available for the repository.

Issues are reserved for reproducible bugs. Do not open an issue for usage help,
provider setup help, roadmap discussion, or feature planning.

When asking for help, include:

- `vue-ai-hooks` version
- Vue version
- Node and package manager versions
- Provider or custom provider adapter
- Minimal code sample or reproduction

## Bugs

Use the bug report template for reproducible defects. Include the smallest code
sample that shows the problem and the exact error message or unexpected output.

## Feature requests

Use GitHub Discussions for new APIs, providers, or behavior changes. If you
already have an implementation, open a PR and describe the problem, proposed API
shape, and migration impact in the PR template.

## Issue activity

Maintainers keep accepted bug issues active by assigning a next action, asking
for missing reproduction details, or posting a status update when the next
action changes. Reporters can help by answering follow-up questions, keeping
minimal reproductions current, and confirming whether a proposed fix solves the
problem.

Issues may be closed when they are duplicates, cannot be reproduced with the
requested details, or stay blocked on missing reporter feedback. Avoid empty
`bump` comments; add new evidence, reproduction updates, or implementation
notes instead.

## Security

Do not open public issues for suspected vulnerabilities, leaked provider keys,
or credential-handling problems. Follow [`SECURITY.md`](./SECURITY.md) instead.
